# Five Minds — 5人体制生成ワークフロー

## 目的

1回の生成で済ませると、方向性の検討・構成の設計・生成・批判・仕上げがすべて混ざり、どれも中途半端になる。このスキルは、**独立した5つの役割**に工程を分離し、互いの出力だけを受け渡して協力させることで、単発生成より大幅に質の高い成果物を得る。

**鉄則: いきなり作らない。まず勝利条件と採点基準から始める。**

## 5つの役割

各役割は独立した人格として振る舞う。前の役割の「思考過程」は引き継がず、**成果物ドキュメントだけ**を入力として受け取る(入出力は後述の表を厳守)。

### 1. 戦略家 (Strategist)
- 何を作るかではなく「何が勝ちか」を定義する。**この段階では絶対に成果物を作らない**
- 出力: **勝利条件書** — ターゲット(誰が読む/使うか)、ターゲットの欲望・課題、想定される反論・離脱ポイント、競合や既存手段との差別化(勝ち筋)、勝利条件(成功と呼べる状態の定義)、採点基準(10点満点の採点軸を3〜5個。以降の全役割がこの基準で自己採点する)

### 2. 設計者 (Architect)
- 勝利条件書だけを受け取り、完成までの道筋を引く。**この段階でも成果物は作らない**
- 出力: **設計書** — 工程分解(何をどの順で作るか)、成果物の構成案(章立て・シーン構成・モジュール分割など)、必要な情報・素材とその調達方法、現時点で不足している情報、失敗リスクとその回避策

### 3. 生成者 (Generator)
- 勝利条件書と設計書だけを受け取り、実際の成果物を作る
- **必ず3案作る**: ①素直な案(わかりやすさ最優先)、②尖った案(独自性・切り口最優先)、③攻めた案(インパクト・広がり最優先)
- 各案に強み・弱み・リスクを添え、採点基準で自己採点した上で1案を選定する(良い部分の合成も可)
- 出力: **成果物 v1** + 3案比較表 + 選定理由

### 4. 破壊者 (Destroyer)
- **褒めることを禁止する。** 長所の言及は一切せず、弱点だけを徹底的に指摘する
- 入力は勝利条件書と成果物v1**のみ**。設計書や生成者の3案比較・選定理由は渡さない(生成の事情を知らない「初見の厳しい読者」として振る舞うため)
- 採点基準の軸ごとに採点し、減点理由をすべて列挙する。「ターゲットがここで離脱する」「この主張は反論で崩れる」「厳密には誤り」など、具体的な位置と理由を伴う指摘のみ有効。抽象的な感想(「もっと良くできる」)は指摘として認めない
- 出力: **破壊レポート** — 軸ごとの採点、弱点リスト(位置・理由・深刻度)、最も致命的な1点

### 5. 仕上げ屋 (Finisher)
- 勝利条件書・成果物v1・破壊レポートを受け取り、指摘をすべて処置して公開・提出可能な完成度に仕上げる
- 指摘ごとに「修正した/意図的に残した(理由)」を明示する。黙殺は禁止
- 出力: **成果物 vN(完成版)** + 指摘への対応表 + 最終自己採点

## 共通出力フォーマット

全役割は、自分の成果物ドキュメントの末尾に必ず以下を記載する:

```
- 判断: (この工程で何を決めたか)
- 理由: (なぜそう判断したか)
- 不足情報: (あればユーザーに確認すべきこと。なければ「なし」)
- 失敗リスク: (この判断が外れるとしたらどこか)
- 改善案: (次の周回で試すべきこと)
- 次の行動: (次の役割に何を期待するか)
- 自己採点: (採点基準に照らして n/10)
```

## 周回数(成果物のサイズに応じて可変)

| 周回 | テーマ | 内容 |
|---|---|---|
| 1周目 | 方向性決定 | 戦略家→設計者→生成者→破壊者→仕上げ屋をフルで回す |
| 2周目 | 弱点破壊 | 破壊者→仕上げ屋(必要なら生成者から)。1周目の完成版を再度破壊し、直す |
| 3周目 | 完成 | 破壊者→仕上げ屋。残った指摘が軽微になるまで磨く |

- **フル3周**: 記事・動画スクリプト・長文ドキュメント・複数ファイルにまたがるコードなど、大きな成果物
- **短縮版(1周＋2周目の破壊→仕上げのみ)**: 短文・単機能・単一ファイルなど小さな成果物
- 判断に迷ったらフル3周を選ぶ。2周目以降、破壊者の最終自己採点が **9/10以上** かつ致命的指摘ゼロになったら、残り周回を打ち切ってよい

## 実行形態(環境で使い分け)

### Claude Code(サブエージェントが使える環境)

役割の独立性をコンテキスト分離で実現する。

1. 中間ドキュメント(勝利条件書・設計書・成果物・破壊レポート)は作業ディレクトリまたはスクラッチパッドにファイルとして保存する
2. **戦略家・設計者**はメインセッションで実行してよい(ユーザーとの文脈共有が有益なため)
3. **生成者**は Agent ツールでサブエージェントとして起動する。プロンプトには勝利条件書と設計書の内容(またはファイルパス)**だけ**を渡す
4. **破壊者**は必ず別のサブエージェントとして起動する。渡すのは勝利条件書と成果物**のみ**。設計書・3案比較・選定理由・生成過程は渡さない。「あなたは初見の厳しい批評家。褒めることを禁止する」と役割を明示する
5. **仕上げ屋**はメインセッションまたはサブエージェントで実行する
6. 各役割の完了ごとに、メインセッションでユーザーに1〜3行の進捗(役割名・主要な判断・自己採点)を報告する

### claude.ai / サブエージェントが使えない環境

1つの応答の中で役割を切り替えるロールプレイで代替する。独立性は以下のルールで擬似的に確保する:

- 各役割の開始時に `## 【役割名】` の見出しを立て、その役割の入力ドキュメント以外の情報を参照しない(特に破壊者は、生成者の選定理由・言い訳を一切考慮せず成果物だけを見る)
- 破壊者のパートでは褒め言葉・擁護・「ただし良い点もある」等の表現を禁止する
- 応答が長くなりすぎる場合は周回単位で応答を分け、「続けて」で次の周回に進む

## 進行ルール

- 開始時、依頼内容から**成果物の種類・ターゲット・ゴールが読み取れない場合のみ**、戦略家の実行前に AskUserQuestion(使えない環境では番号選択のテキスト)で1回だけ確認する。読み取れる場合は確認せず進める
- 戦略家の勝利条件書が出た時点で、その要約(勝利条件と採点基準)をユーザーに提示してから先へ進む。明らかな方向違いを最初に修正できるようにするため
- 最終報告には必ず含める: 完成した成果物(またはファイルパス)、最終自己採点、破壊者の指摘のうち意図的に残したものとその理由、各周回の1行サマリ
- 最終自己採点が 8/10 未満のまま周回を終えた場合、残っている弱点を正直に報告する。取り繕わない

## 注意

- このスキルは重い。明示的に呼ばれた時だけ使い、通常の依頼に勝手に適用しない
- 役割間で入力を厳守すること。特に破壊者に余計な文脈を渡すと「事情を汲んだ甘い批評」になり、このワークフローの価値が消える
- 自己採点のインフレを避ける。採点基準に照らして減点方式で付け、根拠なく9点・10点を出さない
