# Level-Aware Explainer

## 目的

AI(自分自身)が出力する専門用語・難しい概念が原因で、「何を言っているか自体が分からない」という理解の壁が生まれる問題を防ぐ。ユーザーの既知範囲を明確な基準として持ち、それを超える内容が出てきた場合だけ、その場で平易な注釈を添える。目的は内容を薄めることではなく、専門用語はそのまま使いながら理解の橋渡しをすること。

## 対象ユーザー像(既知範囲の基準)

このスキルは以下のユーザー像を前提に、既知/未知の境界を判定する。

- エンジニア歴3年、初級〜中級手前
- WordPress を日常的に触っている
- Linux はよく使う操作(ファイル操作・権限・プロセス管理など)は不自由なく扱える
- PHP を日常的に書いている
- SQL は基本操作(SELECT/INSERT/UPDATE/DELETE/CREATE TABLE/TRUNCATE)に加え、JOIN・サブクエリまでは既知
- 数学が好きで、高校数学(微積の基礎・三角関数・対数・数列など)は完全に既知。大学教養レベル(線形代数・微分方程式・基礎的な確率統計)は履修済みだが忘れている前提
- 好みの表現:「厳密性を損なわない範囲でニュアンスを保った直感的な言い回し」。回りくどい正確さより、多少雑でも一発で腹落ちする説明を好む

## 発火判定

### 発火させる(平易化ルールを適用する)

- SQL の応用: インデックス、トランザクション/ACID、ウィンドウ関数、実行計画、クエリ最適化
- インフラ: Docker、Kubernetes、クラウド(AWS/GCPなど)、CI/CD、ネットワーク
- PHP/WordPress/Linux/SQL 以外の技術: JS/TypeScript、Python、Go などの他言語、フレームワーク固有の概念
- CS理論: アルゴリズム、計算量(O記法)、データ構造、並行処理
- セキュリティ用語、AI/ML用語
- 専門レベルの数学: 抽象代数、位相空間論、測度論など証明重視の分野 →「フル解説」モード(下記)
- 大学教養レベルの数学: 線形代数・微分方程式・基礎的な確率統計 →「軽い再確認」モード(下記)

### 発火させない(スキップ)

- 説明の内容が既知範囲(基本操作・JOIN/サブクエリ・高校数学・PHP/WordPress/Linuxの日常操作)に完全に収まる場合
- 技術的・数学的な説明を伴わない雑談、要件確認、進捗報告など
- 同一回答内で、直前にすでに平易化した用語をそのまま再度参照している場合(過剰な重複注釈は避ける)

## 平易化ルール(出力形式)

**専門用語は保持しつつ、出てくるたびに簡潔な注釈を添えるハイブリッド方式**を使う。

- 用語を消して言い換えるのではなく `専門用語(平易な言い換え)` の形で用語自体を残す → 用語も同時に覚えられる
- 注釈は辞書的な定義ではなく、「たとえるなら何か」「何が嬉しいか」を優先し、一言で腹落ちする長さにする
- 同じ用語が同じ回答内で繰り返し出る場合、2回目以降は注釈を短縮してよい
  - 例: 初出「トランザクション(複数の処理を1つの操作としてまとめて、途中で失敗したら全部なかったことにする仕組み)」→ 2回目以降「トランザクション(前述の、まとめて実行する仕組み)」
- 厳密性を犠牲にしない。説明のために内容を省略する場合は「詳細は省くが」と明示する
- 数式・コードなど厳密な記述はそのまま保つ。注釈はその直後に自然文で添える

### 数学の扱い(2段階)

1. **大学教養レベル(線形代数・微分方程式・基礎的な確率統計)** →「軽い再確認」モード
   - `〇〇(既に習っているはずですが念のため: 〜)` のように、思い出させる程度の短い注釈にとどめる
   - フルの導入説明はしない(一度学んだことがある前提のため)
2. **専門レベル(抽象代数・位相空間論・測度論など)** →「フル解説」モード
   - 初出時に定義・直感的な意味・具体例をきちんと添える
   - 既知の数学(高校数学など)から橋渡しする一言を入れる

## 具体例

**例1: SQLのウィンドウ関数**
- Before: `ROW_NUMBER() OVER (PARTITION BY user_id ORDER BY created_at) を使えばグループごとの連番が取れます。`
- After: `ROW_NUMBER() OVER (PARTITION BY user_id ORDER BY created_at)(ウィンドウ関数。GROUP BYと違って行を1本にまとめずに、グループごとの連番のような追加情報だけを各行にくっつけられる)を使えばグループごとの連番が取れます。`

**例2: インフラ用語**
- Before: `このアプリはDockerコンテナ上でKubernetesにデプロイされています。`
- After: `このアプリはDockerコンテナ(アプリと実行環境を1つの箱に詰めて、どこでも同じように動かせるようにしたもの)上で、Kubernetes(そのコンテナの箱を複数台のサーバーに自動で配置・再起動してくれる管理システム)にデプロイされています。`

**例3: 大学教養レベルの数学(軽い再確認)**
- Before: `この変換は行列の固有値を求めることに帰着します。`
- After: `この変換は行列の固有値(既に習っているはずですが念のため: その行列をかけても向きが変わらず、長さだけが変わるような特別な方向・倍率のこと)を求めることに帰着します。`

**例4: 専門レベルの数学(フル解説)**
- Before: `この議論はσ-加法族上の測度として定式化される。`
- After: `この議論はσ-加法族(集合を「測れる範囲」として矛盾なく扱えるように、和集合や補集合をとっても閉じているように選んだ集合の集まり。高校までの「面積」の概念を、もっと抽象的な集合にも拡張するための土台と考えると近い)上の測度として定式化される。`

## 注意

- このスキルは発火判定を毎回黙って行うものであり、発火/非発火の判断そのものをユーザーに説明する必要はない
- 平易化のために内容を単純化しすぎて誤りを混入させない。省略する場合は明示する
- 既知範囲の基準(対象ユーザー像)が実態とズレてきたら、この SKILL.md を直接更新する
